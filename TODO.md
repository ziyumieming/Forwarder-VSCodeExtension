代码分析模块中采用以LSP为主、以Tree-sitter为补充的技术路线。调研发现：VS Code已通过LSP在后台完成了高度优化的代码分析工作，用于支持符号导航、跳转定义、类型推导等核心功能。这些分析结果本身已经覆盖了本项目所需的大部分结构信息，例如类与函数的定义位置、符号层级关系以及跨文件引用关系。相比自行实现完整的语法与语义分析流程，直接基于VS Code提供的LSP能力获取结构信息，能够显著降低开发复杂度，同时保证分析结果在不同语言环境下的一致性与准确性。
因此，本项目将优先通过VS Code暴露的LSP接口获取符号信息和关系数据，用于构建初始的代码结构模型。在部分LSP无法覆盖或精度不足的场景（如快速识别局部结构、补充轻量级语法信息）下，引入Tree-sitter作为补充手段，利用其高性能的增量解析能力对源码进行快速结构提取，从而在完整性与性能之间取得平衡。
在可视化模块方面，本项目选用VS Code Webview作为展示载体，并采用Cytoscape.js作为核心图形渲染与交互库。环境的选择是Canvas与Webview，前者只支持特定图形的绘制，后者则提供了完整的前端运行环境，能够在插件中嵌入含JavaScript的界面，适合自由定制复杂的交互式可视化内容。Cytoscape.js 是一款成熟的图结构可视化库，原生支持节点与边的抽象建模、自动布局算法以及丰富的交互能力，能够较好地应对函数调用图、类关系图等复杂拓扑结构的展示需求。
相比从零实现图形布局与交互逻辑，该方案可以显著减少工程工作量，使开发重点集中于“展示哪些结构、如何引导理解”等高层设计问题。同时，基于统一的图数据模型，可视化模块能够以视图适配的方式支持不同类型的结构图（如类图、调用图），为后续功能扩展提供良好基础。
在语义理解模块中，本项目选择基于VS Code Language Model API调用大语言模型能力，并引入 LangChain.js 构建流程。原生的VS Code Language Model API 在插件环境中更容易进行集成和权限管理。LangChain.js则是在插件开发的环境下构建由LLM驱动的应用的常规选择。
这里的技术路线避免了模型微调等高成本路径，希望只利用通用模型的能力完成语义理解；同时尝试引入Graph-RAG进行结构化上下文注入来提升语义生成质量，以在工程可行性与理解效果之间取得平衡。


1.需求分析
本项目拟实现的 VS Code 插件基于UML图的可视化，面向“辅助开发者理解项目代码结构”这一核心需求展开，其具体需求如下：
i.	代码结构分析与图形生成能力：这是整个系统的前提，也是插件能够发挥价值的基础功能。插件需要能够对项目中的源代码进行静态分析，提取出类、函数、接口等基本结构信息，以及它们之间的继承、实现和调用关系。在此基础上，插件应支持生成和导出 UML 类图、函数调用图等结构化视图，以直观的图形方式呈现项目的整体结构和主要逻辑关系。
ii.	良好的交互能力：可视化图中的类或函数节点应支持基本的交互操作，例如点击节点即可跳转至对应的源代码位置，或在不同视图之间进行联动切换；同时，插件应支持对特定结构关系的高亮展示，如可以突出显示某一函数的调用链，或围绕某个类展示其相关的依赖关系等。
iii.	可视化图中融合语义信息：对图中的节点提供辅助解释，并与合适的交互结合。如在用户将鼠标悬停在类或函数节点上时，动态展示该节点的语义摘要，说明其主要职责或功能作用；当用户选中两个存在调用关系的函数节点时，插件可以生成一段自然语言描述，用于解释二者之间的调用逻辑和数据流关系；插件还应支持以指定入口函数为起点，对完整执行路径进行分析，成整体流程说明，从而帮助用户理解程序的运行逻辑，而不必逐行追踪代码细节。此处仅举几例，这是区别于传统代码可视化工具的重要功能需求。
iv.	整体问答与分析能力：理论上说，插件可以利用建立起的图结构信息作为上下文，对有关项目整体的问题提供更好的支持（如下图）。这是面向图的检索增强生成（Graph-RAG）的思路，恰好与本课题使用的结构和目的相契合。由于这是较前沿的领域，可以参考的资料内容复杂，计划将本需求作为进阶的可选功能实现。
 
v.	可用性、可扩展性、可移植性：基本要求包括在 VS Code 环境中稳定运行、界面布局清晰、操作方式符合开发者使用习惯等等。另外，经过初步调研与验证，插件中数据流动的控制需要与VS Code的API结合实现，在实现时应当将与API无关的业务逻辑作为服务独立处理，方便日后做功能扩展以及向其他编辑器移植。
综上所述，本课题的需求不仅涵盖了代码结构解析与可视化这一传统功能，还强调通过交互设计与大语言模型能力，提升代码语义层面的可理解性，最终目标是成为一种能够切实降低代码理解成本、辅助开发者快速掌握项目结构的智能工具。
2.功能设计
项目需要实现的模块可以大致按前后端划分到运行在 Extension Host 进程中的后端与运行在 Webview 进程中的前端：
 
3.1插件后端
这是系统的核心控制单元，运行于 Node.js 环境中，具备访问文件系统与调用 VS Code API 的能力，核心是图数据结构的管理模块，几个关键模块介绍如下：
i.	LSP客户端：与各编程语言对应的LSP服务器进行通信，发送解析请求，直接获取函数定义、引用关系、调用位置等结构化信息或者是这些代码在编辑器中的位置信息。这一组件避免了从零实现复杂的语言解析逻辑。
ii.	数据适配层：该模块的数据源就是LSP客户端获取的各类信息。一方面，它根据坐标信息读取源代码内容，供给语义理解模块使用；另一方面，它获取符号信息后对结果进行转化，处理为插件内部定义的中间表示（IR）来屏蔽语言差异，为后续的图结构构建与语义分析提供统一的数据基础。
iii.	图结构管理模块：根据数据适配层提供的统一的IR生成各种图的拓扑结构，专注于维护逻辑层面上的各节点的关系和节点本身的标签信息。该模块（通过视图查询器）向前端提供规范化的数据输出，也为语义理解模块充当知识库。
iv.	语义理解模块：通过 VS Code Language Model API 接入大语言模型服务。该模块会读取图结构管理模块维护的结构化数据作为上下文，根据前端交互控制器产生的交互请求的含义生成语义解释。由于大模型调用存在延迟与配额限制，将会对节点（针对具体类、函数）设计结果缓存机制，在首次加载时请求并记录；对节点关系（主要是调用关系）使用实时请求的方式。
3.2插件前端
前端使用的VS Code Webview本质是一个独立的沙箱化 Chromium 环境，所以可以把系统的可视化呈现与用户交互逻辑放置在这个进程中。具体来说：
i.	视图调度器：根据用户当前的查看需求构造请求并做一些优化工作。结合课题的目标，将类图、调用图、包图列为查看需求的候选项，其中包图和类图构成全局视图，让用户看清大项目由哪些文件夹和核心类组成；调用图用作局部交互视图，用户可以展开某个函数的调用链，聚焦于单个功能的实现路径。优化则包括管理窗口状态、对请求结果做缓存等性能和可操作性上的改进。
ii.	渲染与交互控制器：直接面向用户各种绘制并展示各种图，提供不同的样式或布局策略，以及监听用户在图中的操作行为等。这一部分主要是搭建可用界面，并处理大规模图渲染可能带来的性能问题。
